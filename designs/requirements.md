# Kairos — 需求文档 v2

> καιρός — the decisive moment.
>
> AI 驱动的旅拍后期工具链，从调色后素材到剪辑脚本到达芬奇时间线，以 AI 工作链路为核心。

## 1. 项目概述

Kairos 面向专业旅行摄影师 / 纪录片创作者，解决旅拍后期中最耗时的环节：**素材整理、脚本生成、粗剪编排**。

### 1.1 核心定位

- **Phase 1 为 Node.js 库 + Agent Skill**：无 GUI，通过 Agent 中的脚本 + Skill 实现交互式工作流；后续迁移到 Tauri 桌面应用
- **AI 工作链路为核心**：不是"传统工具 + AI 辅助"，而是 AI 主导编排、用户在关键节点审阅调整
- **与达芬奇深度协作**：通过已有的开源 DaVinci Resolve MCP Server 实现双向通信，Kairos 是规划和管理中心，达芬奇是执行引擎
- **与 Pharos 数据联动**：读取 Pharos（兄弟项目）的分镜行程数据，作为脚本生成的关键输入

### 1.2 核心工作流

用户的完整后期流程如下，Kairos 覆盖加粗部分：

```
原始素材
  → **[Kairos] 辅助调色**：通过 MCP 指挥达芬奇批量调色
       - 按拍摄设备配置识别 S-Log3/D-Log M，批量应用 CST/LUT
       - 自动初级校色（白平衡、曝光归一化）
       - 照片仍在 LR 中人工调色（Kairos 不参与）
       - 延时素材人工处理（Kairos 不参与）
  → **[Kairos] 照片 GPS 写入**：从 GPX 轨迹按时间匹配写入照片 EXIF
  → **[Kairos] 素材管理**：导入调色后素材，提取元数据，匹配 GPS 轨迹
  → **[Kairos] 读取 Pharos 分镜数据**：获取行程规划 + 实际拍摄记录
  → **[Kairos] 脚本生成**：基于素材+分镜+GPS 生成剪辑脚本
  → **[Kairos] 交互式脚本调整**：用户在 Agent 对话中审阅/编辑脚本（Phase 1），后续迁移到 Tauri UI
  → **[Kairos] 粗剪**：通过 MCP 在达芬奇中创建时间线、导入素材、按脚本切片、配字幕
  → **[Kairos] 精剪辅助**：给出精剪建议，辅助用户在达芬奇中精剪
```

### 1.3 与达芬奇的关系

**混合协作模式**：
- 前期（素材管理、脚本生成）在 Kairos 中完成
- 进入剪辑后切换到达芬奇，Kairos 通过 MCP 辅助操作
- Kairos 是"大脑"（规划、决策），达芬奇是"手"（执行调色、剪辑）

### 1.4 设备与格式假设

| 类别 | 设备 | 典型格式 |
|------|------|----------|
| 风光相机 | Sony A7R5 / A7R3 | XAVC-S 4K, S-Log3, RAW 照片 |
| 视频机 | Sony ZV-E1 | XAVC-S-I 4K, S-Log3 |
| 无人机 | DJI Mavic 4 Pro | H.265 4K/5.4K, D-Log M |
| 运动相机 | （预留） | — |
| 音频 | 机内 / 外接麦克风 | WAV / AAC |

照片在成片中的角色：**作为静帧素材插入视频时间线**（Ken Burns 效果等），非独立产出。

## 2. 功能模块

### 2.1 调色辅助（Color Assist）

**目标**：通过 MCP 指挥达芬奇，自动完成视频素材的初级调色。

| 能力 | 说明 |
|------|------|
| Log 格式配置 | 按拍摄设备配置色彩空间（不做自动识别）。默认预设：Sony 机身 = S-Log3，DJI 无人机 = D-Log M，未关联设备 = Rec.709。Ingest 阶段通过 EXIF Make/Model 自动关联设备；也可对单条素材手动覆盖 |
| CST/LUT 批量应用 | 通过 MCP 在达芬奇中为每条素材应用对应的 CST 节点或 LUT |
| 初级校色 | 自动检测并修正白平衡偏移和曝光偏差 |
| 场景一致性 | 同一场景内的片段保持色彩一致 |

**不做**：LR 照片调色、风格迁移调色（Phase 2）、延时素材处理。

### 2.2 素材导入与元数据管理（Ingest）

**目标**：导入调色后的素材，建立结构化的素材数据库。

| 能力 | 说明 |
|------|------|
| 文件扫描 | 递归扫描项目目录，识别视频/照片/音频文件 |
| 元数据提取 | EXIF、XMP、文件名时间戳、帧率、编码格式、色彩空间 |
| GPS 轨迹关联 | 优先使用素材自身内嵌 GPS（尤其 DJI 无人机视频的丰富 metadata）；若素材无内嵌 GPS，再使用项目级 GPX 资源按时间戳匹配坐标。当前项目内正式落点为 `gps/tracks/*.gpx` 与 `gps/merged.json`，Analyze 默认遵循 `embedded GPS > project GPX > manual-itinerary` |
| 照片 GPS 写入 | Sony 相机照片 EXIF 缺少 GPS，从 GPX 轨迹按时间匹配写入 EXIF |
| 逆地理编码 | GPS 坐标 → 地名（省/市/景点），用于脚本生成 |
| 代理文件生成 | Kairos 用 FFmpeg 生成低分辨率代理文件，**主要用于本地视觉模型（CLIP/BLIP）识别分析**，同时可作为后续 GUI 版本的预览素材 |
| 场景检测 | 本地轻量模型（CLIP/BLIP）提取视觉特征 + LLM 总结，将素材分组为"场景" |

**不做**：素材质量评分（Phase 1 用户手动标记）。

### 2.3 Pharos 数据集成

**目标**：读取 Pharos 项目的分镜行程数据，作为脚本生成的核心输入。

| 能力 | 说明 |
|------|------|
| 分镜数据读取 | 读取 Pharos 的可交换分镜 JSON 数据（含分镜类型、描述、设备、优先级） |
| 拍摄记录匹配 | 关联 Pharos 移动端的实际拍摄记录（时间、GPS、完成状态） |
| 素材-分镜关联 | 利用分镜的拍摄时间+GPS 精确匹配素材，而非盲猜 |

**注意**：Pharos 数据交换协议尚在设计中，此模块接口预留，待 Pharos 确定后对接。

### 2.4 脚本生成（Script）

**目标**：基于素材、Pharos 分镜数据、GPS 轨迹、**用户过往成片风格**，生成剪辑脚本。

| 能力 | 说明 |
|------|------|
| 风格参考 | 导入用户过去的成片（MP4），自动分析提取**风格档案**（见下方详述），作为脚本生成的风格约束 |
| 分镜细化 | 在 Pharos 分镜基础上，结合实际素材内容，细化为剪辑脚本 |
| 素材内容推导 | AI 分析视频/照片内容，**用于片头、精彩镜头回顾、花絮等非分镜驱动的叙事段落** |
| GPS 轨迹叙事 | 根据轨迹和时间线构建空间叙事骨架 |
| 多风格支持 | Vlog 口播体 / 纪录片旁白体 / 纯字幕体 / 散文体 |
| 分段输出 | 按场景分段，每段包含：旁白文字、对应素材引用、预估时长、情绪标签 |
| 交互编辑 | 用户在 Agent 对话中审阅/修改段落，AI 自动调整上下文衔接（Phase 1）；后续迁移到 Tauri UI |

#### 风格档案（Style Profile）

用户提供过去的成片 MP4，Kairos 自动分析并生成持久化的**风格档案**，用于指导后续所有脚本生成：

| 分析维度 | 方法 | 提取内容 |
|----------|------|----------|
| 叙事结构 | FFmpeg 场景切换检测 + 关键帧提取 → LLM 分析 | 片头/正片/结尾占比、段落时长分布、B-Roll 插入模式、叙事节奏曲线 |
| 旁白风格 | Whisper 语音识别 → 旁白文本提取 → LLM 分析 | 措辞习惯、语气特征、信息密度、口头禅、叙述人称 |

- 风格档案为项目级或用户级配置，生成一次后可复用
- 用户可手动编辑风格档案（补充偏好或覆盖 AI 分析结果）
- 脚本生成时，风格档案作为 system prompt / few-shot 示例注入 LLM

### 2.5 粗剪与剪辑辅助（Cut）

**目标**：通过 MCP 在达芬奇中创建时间线，按脚本编排素材。

| 能力 | 说明 |
|------|------|
| 时间线创建 | 通过 MCP 在达芬奇中新建项目/时间线 |
| 素材导入 | 通过 MCP 将素材导入达芬奇 Media Pool |
| 脚本切片 | 根据脚本的素材引用和入出点，在时间线上排列片段 |
| 字幕生成 | 根据脚本旁白文字，生成字幕轨道 |
| 照片静帧 | 照片素材以 Ken Burns 效果插入时间线 |
| 精剪建议 | 分析粗剪结果，给出节奏/转场/B-Roll 建议，辅助用户在达芬奇中精剪 |

**不做**：在 Kairos 内实现时间线编辑器（达芬奇已经是最好的剪辑工具）。

### 2.6 项目管理

| 能力 | 说明 |
|------|------|
| 项目创建 | 一次旅行 = 一个项目 |
| 项目目录结构 | 统一目录规范已包含项目级 GPS 资源目录：`gps/tracks/`（原始轨迹）与 `gps/merged.json`（归一化缓存） |
| 阶段管理 | Color → Ingest → Script → Cut，每阶段可独立进入/回退 |
| 版本管理 | 脚本和时间线的版本快照 |

## 3. 技术栈

### 3.0 Phase 1（Node.js 库 + Agent Skill）

| 层级 | 选型 | 理由 |
|------|------|------|
| 运行环境 | **Node.js + TypeScript** | 快速迭代，Agent 原生支持 |
| 交互层 | **Agent Skill** | Phase 1 无 GUI，通过 Skill 实现交互式工作流 |
| 视频处理 | **FFmpeg**（child_process 调用） | 代理生成、元数据提取、关键帧提取 |
| AI 推理（本地） | **CLIP/BLIP** via ONNX Runtime（Node.js 绑定） | 场景检测的视觉特征提取，分析代理文件 |
| LLM 能力 | **Agent 宿主提供** | 脚本生成、场景总结、精剪建议等（Phase 1 不自行对接云端大模型） |
| 达芬奇集成 | **DaVinci Resolve MCP Server**（已有开源） | 调色、时间线操作 |

### 3.1 后续迁移（Tauri 桌面应用）

| 层级 | 选型 | 理由 |
|------|------|------|
| 桌面框架 | **Tauri v2** | 轻量、Rust 后端性能好、原生体验 |
| 前端 | **React + TypeScript** | 用户最熟悉 |
| 核心逻辑 | 复用 Phase 1 的 Node.js 库 | Tauri 可嵌入 Node.js sidecar |
| 视频预览 | 应用内播放代理文件 | 素材浏览和脚本关联预览 |

### 3.2 AI 能力架构

- Phase 1 LLM 能力由 Agent 宿主提供，Kairos 不自行对接云端大模型
- Kairos Core 管理本地小模型（CLIP/BLIP 视觉特征 + Whisper 语音识别）
- Skill 层通过 prompt 模板 + 结构化数据交换驱动 Agent LLM
- Phase 2 迁移 Tauri 后引入独立 AI Provider 接口（OpenAI / Anthropic / Ollama）

## 4. 阶段规划

### Phase 1 — MVP（Node.js 库 + Agent Skill）

核心目标：跑通从素材到达芬奇时间线的完整链路，通过 Agent Skill 交互式操作。

- [ ] Node.js 核心库骨架 + Agent Skill 定义
- [ ] 素材扫描、元数据提取、GPS 轨迹关联
- [ ] 照片 GPS 写入（GPX → EXIF）
- [ ] 代理文件生成（FFmpeg），用于本地模型分析
- [ ] 场景检测（CLIP/BLIP 本地分析代理文件 + LLM 总结）
- [ ] Pharos 分镜数据读取（接口预留）
- [ ] 脚本生成（LLM，基于素材+分镜+GPS+素材内容理解）
- [ ] Agent 中交互式脚本审阅/编辑
- [ ] 达芬奇 MCP 集成：时间线创建 + 素材导入 + 切片 + 字幕
- [ ] 达芬奇 MCP 调色辅助：Log 文件夹配置 + CST 应用 + 初级校色

### Phase 2 — Tauri 桌面应用 + 深化

- [ ] 迁移到 Tauri + React 桌面应用
- [ ] 应用内视频代理预览
- [ ] 交互式脚本编辑 UI
- [ ] 精剪建议引擎
- [ ] 风格迁移调色
- [ ] B-Roll 自动识别与插入

### Phase 3 — 打磨

- [ ] 音乐适配
- [ ] 多风格脚本
- [ ] 地图轨迹可视化
- [ ] 项目导出/归档
- [ ] 离线 LLM 支持
- [ ] DaVinci Resolve PowerGrade 导出

## 5. 非功能性需求

| 维度 | 要求 |
|------|------|
| 性能 | 500 条 4K 素材的元数据提取 < 10 分钟（含缩略图生成） |
| 可扩展性 | AI Provider 插件化，方便添加新模型 |
| 数据安全 | 素材文件只读引用（除照片 GPS 写入外），不移动/修改原始文件 |
| 离线可用 | 元数据提取、代理生成、场景检测可离线；脚本生成需要网络（或本地 LLM） |
| 跨平台 | Node.js 天然跨平台；macOS（开发机）+ Windows（性能机）并重 |
| 可迁移性 | Phase 1 核心逻辑以库形式封装，便于后续迁移到 Tauri 桌面应用 |
