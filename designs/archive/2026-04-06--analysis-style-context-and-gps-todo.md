# Analyze / Style / Context / GPS Todo

## 背景

这次新西兰 Intro 试跑暴露出来的问题，不只是单次生成失误，而是 Analyze、Style Analysis、Script、Timeline 之间的中间表示和消费方式都还不够稳。

这份文档先把当前已经明确的系统性问题收口，后续按专项逐步优化。

## 目标

- 让复杂旅程项目的素材分析结果更接近创作者真正需要的创作语言。
- 让 style analysis 输出从“长文总结”变成“可直接消费的功能骨架 + 细节语法”。
- 让强模型吃到更干净、更短、更有优先级的输入，减少上下文污染和幻觉。
- 在 Pharos 接入前先正视 GPS 信息缺口；接入后把空间真值正式补到 Analyze 主链里。
- 为后续“先人工剪视频，再从剪映草稿生成字幕”建立可复用的读取路径。

## 现阶段判断

- 当前问题已经不能简单归因到 `4B`。即使主生成阶段换成 `GPT-5.4`，结果仍然说明“任务形态 + 中间表示 + 上下文组织”才是更核心的问题。
- 对复杂旅程来说，真正缺的是更好的结构化前置层，而不是单纯继续堆长 prompt。
- GPS 缺口会持续影响空间理解、地理重置、路线推进和地点介绍相关判断；这不是小瑕疵，而是基础信息缺失。

## Todo

### 1. 模型分层与任务重新分工

- [ ] 重新定义每类模型负责的任务边界，不再默认让单个主模型承担“全量理解 + 全量生成 + 全量约束对齐”。
- [ ] 把复杂旅程的关键决策拆成更小的正式子任务：
  - 骨架抽取
  - 素材标签增强
  - recall 压缩
  - outline / script 生成
  - timeline 对齐
- [ ] 为线上模型接入准备一个小项目验证方案，先测素材分析阶段，不直接在大项目上全量切换。
- [ ] 区分“需要更强模型”的任务和“可以由轻模型或规则完成”的任务，避免高成本模型做低价值工作。

### 2. Style Analysis 改成“功能骨架 -> 块内语法 -> 细节偏好”

- [ ] 不再把风格分析主要写成“语气 + 节奏印象”的长文。
- [ ] 先抽功能骨架，再抽每个骨架块内部的素材语法和镜头语法。
- [ ] 骨架块边界按功能需求划分，而不是按常见时长模板硬切。
- [ ] 功能骨架候选包括但不限于：
  - 入场 / 航拍引入
  - 人物 / 景点介绍合集
  - 时间空间尺度的行车合集
  - 航拍地点介绍合集
  - 冲突事件合集
  - 延时摄影集群
  - 地理重置
  - 情绪释放
- [ ] 每个块至少要补：
  - 它解决什么叙事任务
  - 常见素材组合
  - 常见运镜
  - 常见转场作用
  - 哪些语法适合它，哪些不适合
- [ ] 明确原声窗口、地点切换、尺度切换、hero material 充足度，都是决定块边界的真实约束。

### 3. 素材分析补“镜头语法层”和“创作角色层”

- [ ] 当前 `aerial / drive / broll / talking-head / timelapse` 作为主类型保留，但需要新增更细的标签。
- [ ] 增加运镜方式标签：
  - 推进
  - 拉远
  - 横移
  - 俯冲
  - 跟车
  - 固定机位延时
  - 手持观察
  - 前挡 / 车内驾驶视角
- [ ] 增加视角和讲述角色标签：
  - 第一人称景点位置介绍
  - 第三人称景点介绍
  - 开车出发自拍
  - 开车行驶自拍
  - 车内人物反应
- [ ] 增加功能角色标签：
  - 开场建场
  - 地理重置
  - 路线推进
  - 情绪释放
  - 冲突提示
  - 信息交代
- [ ] 增加地点和空间层级标签：
  - 地点名
  - 区域层级
  - 是否适合做尺度建立
- [ ] 增加人物状态标签：
  - 出发
  - 上路
  - 到达
  - 停车
  - 故障
  - 解释
  - 感叹
- [ ] 评估哪些标签适合自动生成，哪些更适合人工补注或半自动校对。

### 4. Prompt / Context 合同瘦身与优先级重排

- [ ] 给主生成链路定义明确的上下文优先级：
  - `P0` 硬约束
  - `P1` 当前任务合同
  - `P2` 功能骨架 / beat skeleton
  - `P3` 压缩后的候选素材摘要
  - `P4` 风格参数
  - `P5` 长文解释
- [ ] 长文档不再作为主输入，只保留辅助说明作用。
- [ ] 清理当前“信息很多但同层级混堆”的输入组织方式。
- [ ] 为 Script / Timeline 前增加一个 `clean briefing layer`，只保留当前回合真正需要的高优先级信息。
- [ ] 避免把完整 `slices.json`、长历史讨论、长风格文稿一起直接塞给主模型。

### 5. 引入 subagent 做前置上下文隔离

- [ ] 把 subagent 的主要价值定义为“前置上下文去污染”，不只是并行提速。
- [ ] 典型拆分方式：
  - 一个 subagent 只做风格骨架抽取
  - 一个 subagent 只做素材标签增强
  - 一个 subagent 只做 recall 候选压缩
  - 主 agent 只做最终整合
- [ ] 明确主 agent 默认不直接吞全部原始中间产物，而是优先消费 subagent 压缩结果。
- [ ] 为这些子任务建立稳定输出格式，避免不同 agent 之间继续互相污染上下文。

### 6. GPS / 空间真值补强

- [ ] 把“当前 GPS 大量依赖推测”正式记为 Analyze 阶段的重要信息缺口。
- [ ] 在 Pharos 接入前，继续保留当前 `embedded GPS > project GPX > derived-track > none` 的优先级，但要更明确区分：
  - 真值
  - 高可信推断
  - 低可信推断
- [ ] 在 coarse report / slice / chronology 中显式暴露空间置信度，不要让推断结果看起来像真值。
- [ ] Pharos 接入后，把更可靠的空间与轨迹信息并入 Analyze 主链，优先服务：
  - 地理重置
  - 路线推进
  - 航拍地点确认
  - 景点介绍
- [ ] 重新评估有了 Pharos 之后，哪些地点判断、路线骨架和 recall 策略可以由“推测”切换为“真值驱动”。

### 7. 剪映草稿读取与字幕工作流

- [ ] 建立“用户先在剪映里手剪视频，Kairos 再读取 draft 生成字幕”的正式工作流。
- [ ] 明确当前至少要能稳定读取：
  - 视频轨片段时间线
  - 文字轨 / 字幕轨
  - 片段时长
  - 草稿分辨率 / 帧率
- [ ] 评估读取来源优先级：
  - `draft_info.json`
  - 其他可读 companion files
  - 是否必须理解 `draft_content.json` 的编码 / 压缩格式
- [ ] 在这个工作流里把“字幕生成”定义成基于已剪视频节奏的任务，而不是重新生成视频结构。

## 风险

- 如果只继续扩写文档而不升级结构化输出，强模型还是会在长上下文里产生幻觉。
- 如果功能骨架被误写成固定模板，会压平创作空间。
- 如果素材标签太多但没有主次，Analyze 会变慢，且下游还是不知道该先看什么。
- 如果 GPS 推断结果不区分置信度，会继续在地理和路线判断上误导下游。

## Done Criteria

- 复杂旅程项目可以先得到稳定的功能骨架，再做文稿和时间线生成。
- style analysis 的结果可以直接指导 recall / outline / intro / montage，而不是依赖模型二次猜。
- 素材分析结果能表达“这是怎样的镜头语法”和“它在叙事里能干什么”，而不只是粗分类。
- 主生成链路的 prompt 明显更短、更干净、更有优先级。
- GPS 缺口在 Pharos 接入前被明确标注，在接入后被正式补强。
- 剪映草稿读取可以支撑“视频人工剪，字幕由 Kairos 生成”的协作流程。
