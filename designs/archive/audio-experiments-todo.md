# Audio Experiments Todo

## Goal

单独记录 Kairos 后续与音频相关的实验事项，避免和当前主线开发混在一起遗忘。当前先拆成两条独立方向：

- 自动配乐
- 配音 / TTS / 声音克隆调研

## 1. 自动配乐

- [ ] 验证现有音乐库接入路径：是否要扩展现有 ingest 直接纳入 `audio` 资产，还是单独做音乐库索引。
- [ ] 先做 `library-first` 自动配乐实验：优先从现有音乐库中按情绪、节奏、BPM、是否纯音乐来选曲。
- [ ] 为音乐素材整理最小元数据集：`mood`、`energy`、`bpm`、`instrumental/vocal`、`license/source`、`duration`。
- [ ] 设计基于视频内容和 style profile 的选曲输入：段落角色、节奏抬升点、航拍/转场/照片留白等。
- [ ] 做一个最小时间线实验：给现有视频自动铺一条 `music` 音轨，并处理淡入淡出。
- [ ] 评估旁白 / 原声 / 配乐三者冲突：什么时候需要 ducking，什么时候应完全让配乐退出。
- [ ] 验证导出可编辑性：剪映草稿中的音乐轨是否方便后续手工调整。

## 2. 生成式配乐

- [ ] 做 `Suno` 类 API 可行性实验，但先只当实验路线，不直接作为主生产链路。
- [ ] 单独核实 `Suno` 的开发者接入方式、API 稳定性、商业授权和可持续性，不把第三方封装误当官方平台。
- [ ] 对比更工程化的候选：`Mubert API`、`Google Lyria 3`，重点看授权清晰度和 developer support。
- [ ] 生成式配乐实验默认限制为 `instrumental / background score`，避免歌词和旁白冲突。
- [ ] 尝试用视频摘要 / segment brief / style rhythm grammar 自动生成配乐 prompt。
- [ ] 评估 hybrid 策略：`library-first`，找不到合适曲目时才 fallback 到生成式配乐。

## 3. 配音 / TTS / 声音克隆

- [ ] 回收并整理之前的配音调研结论：`Azure TTS`、`MiniMax TTS`、声音克隆、部署成本、效果差异。
- [ ] 补一版中文场景下的对比：自然度、情绪控制、长文本稳定性、价格、部署复杂度。
- [ ] 如果继续做实验，优先做最小闭环：`脚本文本 -> 生成旁白 -> 对齐字幕/时间线`。
- [ ] 单独评估是否需要“你的声音克隆”这条线；若做，先明确成本、数据需求和隐私边界。
- [ ] 保持和当前 `speech-paced subtitles` 路线解耦，不要在同一轮把配音能力强绑进主时间线逻辑。

## 4. 当前判断

- 近期更值得优先实验的是：`现有音乐库自动配乐`
- 生成式配乐适合作为第二阶段或 fallback 路线
- 配音 / TTS 继续保留为独立研究项，等自动配乐方向明确后再决定是否推进实现
